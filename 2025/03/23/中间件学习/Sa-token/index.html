为什么要用 Sa-token

需求分析

解决需求：同端登录冲突检测

在多用户系统中，如电商平台、企业办公系统或社交应用，用户的账户安全和系统的正常使用很重要。

如果同一个账户同时在多个设备上登录，可能造成数据泄露或账户盗用的情况，也可能发生数据不一致，如订单重复提交、聊天记录不一致等情况。

为了防止这些情况，我们的系统需要能够实时监控和检测同一账户在多个设备上的登录情况，在检测到冲突时，及时通知用户并采取相应的安全措施（如强制下线或警告）。

技术方案选型

业务中常见的三种冲突检测策略：

  1. 单点登录模式：同一时间，只允许同一账户在一个设备上登录。即每次新设备登录时，检测当前账户是否已经在其他设备上登录，若已登录则将其他设备强制下线。一般应用于包含敏感数据，同一时间只能被一个使用者操作一个账号的场景
  2. 多设备登录限制：允许同一个账户在多个设备上登录，但是有设备数量限制（如视频会员最多允许两台电视登录），每次登录时，检测当前账户已登录设备数量，若超出限制，系统阻止登录或强制最早登录的设备下线。
  3. 同设备类型限制（同端互斥登录）：允许同一账户在不同设备类型（手机和 pc）上同时登录，但同一类型只能登录一个。例如，用户可同时在电脑和 iPad 上使用我们的平台，但不能在两台手机上登录在线。

同端互斥登录实现思路：

  1. 用户登录时获取当前设备信息（通过 user-agent 获取）
  2. 将用户登录信息与设备信息一起保存（本地/第三方缓存中）
  3. 用户登录 时判断同设备是否已经登录（本地/缓存中是否存在）
  4. 如果检测到冲突，可以直接顶号（将较早登录的设备下线，即移除登录态）

采用 sa-token 就能够快且安全的实现第三种冲突检测策略，

Sa-Token 入门

Sa-Token 是一个轻量级 Java 权限认证框架，主要解决：登录认证、权限认证、单点登录、OAuth2.0、分布式Session会话、微服务网关鉴权等一系列权限相关问题。

Sa-Token 功能一览：

登录认证

设计思路

对于需要登录才可以访问的接口，增加一层接口进行校验，如果校验通过则正常返回数据，如果校验不通过则抛出异常通知其必须先登录。

登录访问流程：

用户提交 name + pwd 参数调用登录接口

登录成功返回这个用户的 token 会话凭证

每次登录都会携带这个 token

服务器根据这个 token 判断此次会话是否登录成功

登录与注销

只此一句代码，便可以使会话登录成功，实际上，Sa-Token 在背后做了大量的工作，包括但不限于：

  1. 检查此账号是否之前已有登录；
  2. 为账号生成 Token 凭证与 Session 会话；
  3. 记录 Token 活跃时间；
  4. 通知全局侦听器，xx 账号登录成功；
  5. 将 Token 注入到请求上下文；
  6. 等等其它工作……

注解鉴权

我们现在的项目是使用 AOP + 自定义权限校验注解来实现的登录认证和权限认证，全局拦截器完成注解鉴权功能

首先将 sa-token 的拦截器注册到容器当中

之后就可以顺利使用注解进行鉴权

Sa-Token 还支持另外一种鉴权模式，通过编写 路由拦截代码实现鉴权，可以实现更灵活、集中式的鉴权能力。

用户的每次登录，服务端都会建立一个基于 用户 id 生成一个 session 以及一个与当前客户端的 token，当有新的登录操作发送时，会检测该用户 session 对应的 val 当中是否有对应设备的 token，如果有那么就会把旧的 token 状态设置为 -4，而后让当前登录生效，重新生成yi’ge
