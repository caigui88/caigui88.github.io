Github Action

前言

为了避免每次项目更新都进行手动的：打包、关闭旧版本程序和启动新版本程序等步骤，使用github提供的github-action来完成上述的过程的自动化实现。

什么是 Gitub-Actions？

GitHub Actions 是自动化工作流程工具，用于自动化构建、测试、部署和集成软件项目。它通过一系列事件、触发器和任务来执行各种操作，以实现持续集成和持续交付。

GitHub 允许你可以发布以及共享你自己的 Actions，别人可以通过市场找到你的 Actions 从而不需要在重复造轮子了。

什么是 CI/CD？

CI/CD 是持续集成（Continuous Integration）与持续交付 / 持续部署（Continuous Delivery/Continuous Deployment）的缩写，是一种软件开发实践和方法论。

持续集成（CI）是指开发人员将其代码频繁地集成到共享代码存储库中，并通过自动化构建和测试流程来验证代码的正确性。这样可以确保团队成员的代码变更能够快速地集成到主干代码中，并减少代码集成过程中的错误。

持续交付（CD）是指将通过持续集成产生的软件交付给质量保证团队进行进一步的自动化测试、部署和发布。持续交付的目标是确保软件始终处于可部署状态，以便可以随时交付给用户。

持续部署（CD）是持续交付的进一步延伸，指将通过持续交付阶段产生的可部署软件自动地部署到生产环境中，使软件能够快速地、可靠地投入使用。

通过采用 CI/CD 实践，团队可以实现更快速、可靠、高质量的软件交付。自动化的构建、测试、部署和发布流程能够减少人为错误，提高开发团队的生产效率，并使软件交付过程更加可靠和可追踪。

以上为概念性知识点，对github-action的原理有一个大概的了解，方便更好的理解实际脚本代码中的运行逻辑和含义。

术语

一些基础术语

  1. workflow （工作流程）： 持续集成一次运行的过程，就是一个 workflow。

    1. GitHub Actions 的配置文件叫做 workflow 文件，存放在代码仓库的.github/workflows目录

    2. 可以采用yaml格式文件作为 workflow 文件，一个

      库中可以有多个 workflow 文件，会各自按照对应的event条件执行。

      只要github发现有yml文件，且满足event条件，就会自动运行。

  2. Event （事件）：触发工作流运行的动作，例如每次的push、pull_request等

  3. job （任务） ：每一个job都是工作流中独立的单元，

  4. step（步骤）：每个 job 由多个 step 构成，一步步完成。

  5. action （动作）：每个 step 可以依次执行一个或多个命令（action）。

本项目的部署

流程分析

本项目后端是以SpringBoot为框架搭建的单体架构，因此对于自动化部署的操作可以分解为：

  1. 项目在本地修改成功后push到origin仓库
  2. 参与者在origin仓库进行pull_request操作，更新upstream仓库的指定分支
  3. 使用github action对于更新后的项目进行Maven构建，生成jar包
  4. 将jar包添加到入docker镜像，并设置好运行规则
  5. 将docker上传至个人的阿里云docker镜像源仓库
  6. 服务器从项目docker镜像源处拉取docker镜像
  7. 将拉取的镜像运行，指定端口号为8121

规范化操作：

PR 操作流程

Fork 库并创建分支

  • 每位贡献者从主库 release/test 分支 fork 项目至个人仓库，并创建新的开发分支（例如 feature/xxx 或 bugfix/xxx），确保开发分支命名规范，便于识别。

本地开发并提交代码

  • 在本地进行开发、调试，确保提交内容符合代码规范。

  • 提交时附上清晰的 commit message，例如：

Pull Request 提交

  • 当完成开发或修复后，将代码 push 到 fork 仓库，并向主库的 release/test 分支提交 PR（Pull Request）。
  • 在提交 PR 时，请确保以下几点：
    • PR 标题清晰简明，包含任务编号（如适用），如 feat: 添加登录接口 (#123)
    • PR 描述中包含详细的变更内容、相关 issue 链接、影响模块和测试策略等。
    • 在提交 PR 之前，请确保已在本地或 CI 环境中通过所有测试。

PR 审核流程

  • 团队成员或维护者对 PR 进行审核，确保代码质量和功能正确。
  • 审核通过后，PR 合并至 release/test 分支。

在pr审核通过之后就会启动工作流以实现项目更新后的构建和服务器部署。
