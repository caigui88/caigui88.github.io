JavaWeb学习经验总结

本文档用于总结学习黑马的javaweb课程的知识点和内容，根据视频顺序进行总结

项目总览

总览本项目，是一个实现了前后端发分离的单体架构项目。

单体架构：即一个项目所需的所有功能组件：包括业务逻辑，功能和交互组件都集成在一个环境当中，最终上限部署时打包在一个war包当中，并运行在一个服务器上。

经典的单体架构技术选择为：Java➕Spring MVC

此处引出黑马javaweb选择的框架：SSM框架。

SSM框架即为：Spring➕Spring MVC➕Mybatis，是一套标准的MVC模式：Model-View-Controller模式，模型、视图与控制器。

MVC模式三大部分：

  1. **Model 模型：**即实际功能与业务逻辑的实现区域，同时具备对数据的直接访问权。
  2. **Controller 控制器：**负责进行事件的处理并作出事件的响应。
  3. **View 视图：**提供给用户的可直观使用的交互场所，view可以通过用户的行为向控制器进行事件的传递并通过控制器间接与model进行交互

MVC的优缺：

  1. 低耦合度：每一层都职责分明，各层间采取接口的形式进行交互，某一层的改变不会影响其他层
  2. 复用性高：低耦合其中一个优势就是复用性的提高，例如逻辑上的一个功能模块：支付，既可以具体实现为微信支付，也可以具体实现为支付宝支付、银行卡支付……

**SSM框架各大部件：**a

  1. **持久层：**Dao(mapper)层

    1. 对数据库具体的操作语句放在这一层

  2. **业务层：**Service层

    1. Service负责实际的功能和业务逻辑，通过定义XXX_Service接口和对应的XXX_ServiceImplement来实现具体的功能

    2. Service层向下直接和Dao层进行对接，

      向上直接和Controller对接

  3. **表现层：**Controller(header)层

    1. 负责具体的业务控制流程，即控制业务功能的调用

  4. **视图层：**View层

    1. 与Controller层对接，负责将前端接收到的信号传递给Controller以完成对业务功能的调用。

Spring：

Spring中最重要的几个概念：

DI(依赖注入):

由spring框架主动创建被调用类的对象（存储在IOC容器当中），并将创建出的对象注入到我们所声明的对象当中，在用户的视角里即，一个注解➕一个为类对象的声明即可调用一个未实例化的类对象。

IOC(控制反转)：

控制反转即控制权的反转，这里的控制权指的是仅需调用某个类对象当中的方法而不使用该类对象本身时，将该类对象的生命周期的管理控制权，通过IOC将该控制权交给第三方，由第三方进行统一调度管理。

通过注解的方式，使用单例模式，为创建出的类对象进行装载

AOP(切面编程)：

在不改变原本代码模型的基础上，通过动态代理的技术，增加业务功能以满足需求。

SpringBoot：

基于Spring而进行升级的全新框架

核心组件：

自动配置：

会自动根据添加的jar依赖，进行Spring应用程序合理的默认配置

起步依赖：

将常见的库整合起来，而开发时仅仅需要引入一个“spring-xx-xx-starter”起步依赖，即可导入依赖，避免了不同依赖间版本冲突和引入大量依赖的繁琐。

Mybatis：

核心是SqlSession，Mybatis是对jdbc的封装，使得对于数据库的操作更加简单清晰。

前端部分

前端部分采用经典的模式：HTML➕CSS➕JavaScript➕Vue

  1. HTML负责网页的整体框架、元素与内容
  2. CSS负责元素的样式、外观
  3. JavaScript负责网页的行为，例如用户点击登录之后将登录信息打包以Json的形式发送到后端。

通过编写HTML语句，将编写好的代码交给浏览器内核core进行处理，最终呈现出用户可使用的可视化网页。

但是现今市面上的浏览器内核各不相同，对于某些技术的支持也参差不齐，因此会出现同一段代码在不同浏览器当中呈现的效果不同的问题。

HTML：

超文本标记语言：通过使用各种不同的标签，在可以显示文字的基础上还可以定义图片、音频、视频等内容。

CSS：

Cascading style sheet：层叠样式表，用于控制html页面的样式。

后端部分

Maven

一款用于管理和构建Java项目的工具，基于项目对象模型(Project Object Model，简称POM)

可以做到：

  1. 依赖的统一管理：

    1. 仅需在pom文件中加入标签即可自动导入需要的依赖

  2. 项目结构的统一构建

    1. 使用maven工具构建项目，得到的项目都具有同样的目录结构

Maven模型：

  • **项目对象模型：**Project Object Model，将每一个模块和项目抽象层一个对象模型，具备自己专属的坐标——jar包的唯一标识，用于定位资源的位置。
  • **依赖管理模型：**Dependency
  • **构建生命周期/阶段：**Build lifecycle & phases

Maven坐标：

  1. maven中的坐标是资源的唯一标识，通过该坐标可以唯一定位资源位置
  2. 使用坐标来定义或引入项目中需要的依赖
  • Maven坐标的组成：
    • groupId：定义当前Maven项目隶属组织名称（一般为域名反写，如：com.itheima）
    • artifactId：定义当前Maven项目名称（通常是模块名称）
    • version：定义当前项目版本号

依赖管理

依赖配置

添加dependencies标签后，

使用dependency引入依赖坐标groupId，artifactId，version

依赖传递

依赖会形成像子父类继承一样的关系，即具有传递性

  • 直接依赖：在当前项目中通过依赖配置建立的依赖关系
  • 间接依赖：被依赖的资源如果配置了其他的依赖资源，这当前项目也会间接依赖其他资源

排除依赖：也可以主动断开依赖的资源，被排除的依赖无需指定版本

依赖范围

依赖的默认使用范围是全局使用，但也可以通过<scope> </scope>标签来设置其作用范围。

作用范围：

  • 主程序范围有效（main文件夹范围内）
  • 测试程序范围有效（test文件夹范围内）
  • 是否参与打包运行（package指令范围内）

生命周期

三套独立的生命周期

clean：清理工作，移除上一次构建生成的文件

default：

  • 核心工作，如：编译、测试、打包、安装、部署等
  • compile：编译项目源代码
  • test：使用合适的单元测试框架运行测试（junit）
  • packag：将编译后的文件打包，如jar和war
  • install：安装项目到本地仓库

site：生成报告、发布站点等

后端使用技术概况

  1. IOC Inversion of Control 控制权反转：（是一种设计思想，而非具体的技术）

    • 在传统的Java程序中，一个对象的生成和回收的控制权掌握在程序员手中，程序员会依据所需去创建对象

      但是在spring框架中，一个对象的创建的控制权交给框架实现，由IOC容器统一管理这些对象的生命周期和对象间的关系，从而实现所谓的控制权反转

      使用IOC将原本组件间直接的依赖关系转换成组件——框架——组件的间接依赖关系，并由容器动态管理，进行了组件间的解耦

    • 示例：

      • public class FXNewsProvider {
   private IFXNewsListener newsListener = new DowJonesNewsListener();
   private IFXNewsPersister newPersistener = new DowJonesNewsPersister();
   public void getAndPersistNews() {
      String[] newsIds = newsListener.getAvailableNewsIds();
      if(ArrayUtils.isEmpty(newsIds)) {
         return;
      }
      for(String newsId : newsIds) {
         FXNewsBean newsBean = newsListener.getNewsByPK(newsId);
         newPersistener.persistNews(newsBean);
         newsListener.postProcessIfNecessary(newsId);
      }
   }
}

      • 在Java中，由于只存在类的概念而没有单独的函数，因此当我们需要调用某一个方法是通常需要去先创建一个对象，而后通过该对象进行方法的调用。

        但是本质上我们需要的只是调用方法，对象是不得已才创建的，因此只要是一个可以调用方法的对象，那么无所谓是我们创建的还是其他地方创建的

        在这种背景下，我们通过使用IOC容器，将对象的创建交给框架，框架只需要在我们用到这个对象时完成依赖注入，使对象可以正常使用即可

  2. DI Dependency Injection 依赖注入：

    • DI是IOC的一种实现方式，通过将对象的依赖关系注入到对象中；而不是由对象自己创建对象，形成依赖关系——即：组件之间依赖关系由容器在运行期间根据各种注解决定，形象的说，即由容器动态的将某个依赖关系注入到组件之中。
    • 提升组件重用的频率，并为系统搭建一个灵活、可扩展的平台。

  3. Spring AOP Aspect-Oriented Programming

    Spring的另一大核心，面向切面编程，对OOP的一种补充

    1. 一种编程范式，通过在程序中插入一个切入点，在不修改源代码的前提下给程序动态统一的添加额外的功能
    2. 提供的几种类型的增强：
      1. 前置增强(Before Advice)：
      2. 后置增强(After Advice)：
      3. 返回增强(After Returning Advice)：
      4. 异常增强(After Throwing Advice)：
      5. 环绕增强(Around Advice)：
    3. 使用动态代理技术，临时在内存中创建一个AOP对象，并将目标对象的源代码拷贝过来，插入到指定的增强位置中，从而实现额外功能的添加。
    4. 核心概念：
      1. 切面(Aspect)：相当于一个横切关注点的实例，包括了连接点和通知，可以通过配置文件、注解等方式定义切面。
      2. 连接点(Joinpoint)：可以被切面插入的点，最常用的连接点包括方法调用、方法执行过程中的某个时点等
      3. 通知(Advice)：连接点处执行的点，分为多种通知：前置、后置、环绕。
      4. 切点(Poingcut)：用于定义哪些连接点上应用通知。切点通过表达式定义
      5. 织入(Weaving)：将切面应用到目标对象并创建新的代理对象的过程，织入可以在运行时完成，也可以在编译时完成：编译期织入和运行期织入。
      6. 目标对象(Target)：
      7. 代理对象(Proxy)：

  4. Spring AOP的两种动态代理：JDK动态代理和CGLIB动态代理

    1. 基于类型：JDK动态代理是基于接口的代理，CGLIB动态代理是基于类的代理。
    2. 代理对象的创建：JDK通过Java自带的 java.lang.reflect.Proxy 创建代理对象，该对象必须实现一个或多个接口。CGLIB通过字节码生成技术创建代理对象，无需目标类实现接口，直接继承目标类。
    3. JDk运行时需要使用反射，性能较低；CGLIB通过生成字节码，避免反射，性能较优。
    4. JDk只能代理目标对象的接口方法，不能代理其父类中的方法；CGLIB可以代理目标类及其父类中的方法。
    5. JDK动态代理的核心是InvocationHandler接口和Proxy类；如果目标类没有实现接口，那么SpringAOP会选择使用CGLIB来动态代理目标类。
    6. CGLIB是一个代码生成的类库，可以在运动时动态生成某个类的子类，通过继承来实现动态代理，如果类被标记为final，则无法使用CGLIB做动态代理。

  5. 事务管理：

    1. 确保一组操作的原子性，即要么全部成功，一起提交；如果某一项出现问题，则全部回滚，确保数据的一致性。
    2. 在涉及到多个数据库操作或需要保证数据一致性的业务中，事务管理非常重要的

  6. 全局异常处理：

    1. 对后端程序中发生的所有异常都进行统一处理，提高项目的健壮性和可维护性
    2. 可以使用全局异常处理将异常信息进行记录，返回完整的错误信息给前端，避免系统因未处理异常而崩溃

  7. 过滤器：

    1. 拦截请求或响应，对其进行处理或修改
    2. 在请求到达Controller之前或响应返回给客户端之前，可以使用过滤器进行一些预处理或后处理的操作。

  8. 拦截器：

    1. 与过滤器功能基本一致，只不过拦截器更专注于对Controller的方法进行预处理和后处理
    2. 在业务逻辑处理前后执行一些通用的操作，如权限验证、日志记录等。
